# E2E Remote — Remote URL Testing + Report

Scaffold a minimal Playwright project, explore the remote URL via MCP, create tests, execute, and generate dual reports.

All output must be in **繁體中文**.

## Step 1: Validate Input

- If `$ARGUMENTS` provided, use it as the target URL
- If no argument, ask the user for:
  - Target URL (required)
  - Whether login is needed
  - Test depth (single page / multi-page flow)

## Step 2: Load References (MANDATORY — do not skip)

Read these files before proceeding:
- `references/remote-testing.md` — scaffold templates, MCP auth bridging, remote locator strategy
- `references/mcp-discovery.md` — MCP exploration workflow
- `references/error-discrimination.md` — error classification framework
- `references/auth-patterns.md` — authentication patterns (if login needed)

Do NOT proceed to Step 3 without reading the files listed above.

## Step 3: Scaffold Minimal Playwright Project

In the specified directory (default `~/e2e-remote/{domain}/`):
1. Create `package.json` (only `@playwright/test` dependency)
2. Create `playwright.config.ts` (baseURL = target URL, no webServer)
3. Create `tests/pages/RemoteBasePage.ts`
4. Run `pnpm install` and `pnpm exec playwright install chromium`

See `references/remote-testing.md` § Scaffold for templates.

## Step 4: MCP Authentication (if needed)

Per `references/remote-testing.md` § MCP Authentication:
1. `browser_navigate` → login page
2. `browser_fill_form` + `browser_click` → complete login
3. `browser_run_code` → export cookies + localStorage
4. Write `.auth/remote.json` storageState file
5. Update `playwright.config.ts` to use storageState

## Step 5: MCP Exploration

Per `references/remote-testing.md` § MCP Exploration Workflow:
1. `browser_navigate` → target page
2. `browser_snapshot` → get ARIA tree, build page structure map
3. Explore interactive elements: click tabs/dialogs → snapshot → record
4. Form dry-run: fill → submit → verify result → clean up
5. Produce Discovery Report (elements, forms, tables, tabs)

## Step 6: Generate Test Files

1. `{PageName}Page.ts` — POM extending `RemoteBasePage`, using remote locator strategy (`getByRole` > `getByText` > CSS)
2. `{page-name}.spec.ts` — Test scenarios based on exploration results

**Remote mode differences:**
- No `data-testid` injection (no source code access)
- Locator priority: `getByRole` > `getByText` > CSS > `data-testid` (only if site already has them)
- POM extends `RemoteBasePage` (not local `BasePage`)

## Step 7: Execute Tests

```bash
E2E_REPORT_NAME={page-name} pnpm exec playwright test {spec-path}
```

**After execution, apply Error Discrimination:**

IF any test fails:
├── Classify each failure using error-discrimination.md:
│   ├── FORM SUBMISSION + recoverable keyword?
│   │   └── YES → Fix per strategy table → Retry (max 2)
│   │   └── NO → Report FAIL with classification
│   ├── PAGE LOADING error → Report FAIL
│   └── ELEMENT INTERACTION error → Report FAIL
└── Generate report with per-failure classification

## Step 8: Generate Dual Reports

1. **HTML report** — Generated by Playwright in project's report directory
2. **Markdown report** — `{page-name}-test-report.md` per `references/report-template.md`

## Step 9: Completion

Tell the user:

```
✅ 遠端測試已建立並執行完畢。
📄 報告：
   - HTML: {project-dir}/playwright-report/
   - MD: {page-name}-test-report.md
```

If there are failures, include the error classification summary.
